**Таблица 3.1 — Программно-аппаратный стек DriveLine**

**Технология доступа к данным:** Eloquent ORM + сервисный слой (`app/Services/`). Контроллеры и ресурсы MoonShine остаются «тонкими», бизнес-правила сосредоточены в сервисах.

**Архитектура развёртывания** — рисунок 3.1 (`diagrams/architecture.mmd`).

## 3.2 Реализация базы данных

Схема БД создаётся командой `php artisan migrate` на основе миграций в `database/migrations/`. Актуальное описание таблиц — приложение Е.

**Ключевые особенности реализации:**

- перечисления домена в PHP (`App\Enums\*`) с приведением в миграциях;
- мягкое удаление автомобилей (`softDeletes` на `cars`);
- связь сделок и опций через pivot `option_sale`;
- хранение файлов сделок и заявок в JSON-колонках.

**Пример бизнес-правила в коде** — синхронизация статуса автомобиля при продаже (`SailService::syncSellCarStatus`): при статусе сделки «Подготовка» автомобиль переводится в резерв, при «Завершена» — в «Продан», при «Отменена» — возврат «В наличии».

## 3.5 Технология обработки информации

**Сценарий «Продажа автомобиля»:**

1. Менеджер создаёт сделку типа SELL, выбирает клиента и автомобиль со статусом IN_STOCK.
2. `SailService::createSellSail` в транзакции проверяет доступность авто.
3. При статусе «Подготовка» вызывается `syncSellCarStatus` → RESERVED.
4. При «Завершена» → SOLD; `CarService::forgetCarsCache()` обновляет витрину.
5. При «Отменена» → IN_STOCK.

**Фоновые процессы:** `php artisan queue:work` / Horizon обрабатывает `GenerateReportJob` и заявки на кредит.

**Развёртывание** (кратко): `cp .env.example .env`, `make up`, `npm run build` — подробно в приложении И и [README.md](../../README.md).

## 3.6 Тестирование

### 3.6.1 Функциональные тест-кейсы

**Таблица 3.4 — Функциональное тестирование**

| № | Сценарий | Ожидаемый результат |
