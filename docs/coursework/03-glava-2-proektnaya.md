# 2 Проектная часть

Проектная часть описывает решения по вертикали проектирования: информационные потоки, структуру хранилища данных и организацию диалога пользователя с системой DriveLine. Все сущности согласованы с аналитической частью (глава 1) и реализацией (глава 3).

## 2.1 Разработка информационной модели

Информационная модель представлена **диаграммами потоков данных (DFD)** по методологии Гейна–Сарсона.

### 2.1.1 Контекстная DFD

На рисунке 2.1 (исходник `diagrams/dfd-context.mmd`) система **DriveLine** (процесс 0) показана в окружении внешних сущностей:

- **Посетитель** — просматривает каталог без авторизации;
- **Менеджер** — ведёт клиентов, автомобили, сделки, кредитные заявки;
- **Администратор** — управляет пользователями, ролями, формирует отчёты;
- **Внешний API** — источник данных для консольной команды `car:parser`.

### 2.1.2 Декомпозиция первого уровня

На рисунке 2.2 (`diagrams/dfd-level1.mmd`) выделены процессы и накопители:

| Процесс | Назначение | Накопители |
|---------|------------|------------|
| 1. Отобразить каталог | Публичная витрина | D1 `cars` |
| 2. Управлять автомобилями | CRUD склада | D1 |
| 3. Управлять клиентами | CRUD клиентов | D2 `clients` |
| 4. Оформить сделку | Покупка/продажа, опции | D3 `sails`, D1, D2, D4 `options` |
| 5. Обработать кредит | Заявки на кредит | D5 `credit_applications` |
| 6. Сформировать отчёт | XLSX за период | D3, D6 `reports` |
| 7. Импортировать каталог | Парсер API | D1 |

## 2.2 Проектирование информационного хранилища

Целевая модель данных — **реляционная**, СУБД **PostgreSQL 15**, нормальная форма — **3НФ**.

### 2.2.1 Концептуальная модель БД

Концептуальная ER-диаграмма (рисунок 2.3, `diagrams/er-conceptual.mmd`) включает сущности:

- `cars` — автомобили на складе;
- `clients` — покупатели;
- `sails` — сделки (покупка/продажа);
- `options` — дополнительные услуги;
- связь **N:M** `sails`–`options` реализуется таблицей `option_sale`;
- `credit_applications`, `reports`, `users`, `user_roles`, `user_permissions`.

### 2.2.2 Логическая модель БД

**Приведение к 3НФ.**

1. Связь «сделка — опция» N:M устранена введением ассоциативной таблицы `option_sale` (две связи 1:N).
2. Атрибуты клиента, автомобиля и менеджера не дублируются в `sails` — только внешние ключи `client_id`, `car_id`, `user_id`.
3. Права пользователя вынесены в отдельную сущность `user_permissions` (JSON), что исключает повторение матрицы прав для каждой операции.

**Таблица 2.1 — Примеры транзакций**

| Транзакция | Затрагиваемые таблицы | Ограничение целостности |
|------------|----------------------|-------------------------|
| Завершить сделку продажи | `sails`, `cars` | Статус авто → SOLD; сделка COMPLETED |
| Отменить сделку продажи | `sails`, `cars` | Статус авто → IN_STOCK |
| Удалить клиента | `clients`, `credit_applications` | CASCADE по заявкам |
| Удалить сделку | `sails`, `option_sale` | CASCADE по связям опций |
| Сформировать отчёт | `reports`, `sails` | Только COMPLETED за период |

**Первичные и внешние ключи** (фрагмент):

- `sails.id` — PK; `client_id`, `user_id`, `car_id` — FK с `ON DELETE SET NULL` (миграция `create_sails_table`);
- `option_sale` — составной PK (`option_id`, `sail_id`);
- `clients.phone` — UNIQUE.

### 2.2.3 Физическая модель БД

Физическая модель соответствует миграциям Laravel в каталоге `database/migrations/`. Полное описание полей — в **приложении Е**.

**Таблица 2.2 — Фрагмент физической модели (таблица `cars`)**

| Поле | Тип PostgreSQL | Ограничения | Примечание |
|------|----------------|-------------|------------|
| id | BIGSERIAL | PK | Суррогатный ключ |
| mark | VARCHAR | NOT NULL, INDEX | Марка |
| model | VARCHAR | NOT NULL, INDEX | Модель |
| class | VARCHAR | NULL | Класс |
| vin_code | VARCHAR | NOT NULL | VIN |
| year | INTEGER | NOT NULL | Год выпуска |
| price | INTEGER | NOT NULL | Цена, копейки/рубли — по проекту |
| color | ENUM | NOT NULL | `CarColor` |
| type | ENUM | NOT NULL | NEW / USED |
| status | SMALLINT | DEFAULT 1 | `CarStatus`: 1 наличие, 2 продан, 3 резерв |
| mileage | INTEGER | NULL | Пробег (Б/У) |
| preview, images | VARCHAR / JSON | NULL | Медиа |
| deleted_at | TIMESTAMP | NULL | Soft delete |

**Эволюция модели:** ранее использовалось поле `count` (количество экземпляров); заменено на `status` (миграция `add_cars_status`) — одна запись соответствует одному физическому автомобилю.

**Защита данных на уровне приложения:**

- роли: `user_roles` (администратор `id=1`, менеджер `id=2`);
- детальные права: JSON в `user_permissions.permissions` по ресурсам MoonShine;
- суперпользователь (`isSuperUser`) — полный доступ к отчётам и пользователям (`MoonShineServiceProvider`).

## 2.3 Разработка сценария диалога системы с пользователем

### 2.3.1 Выбор структуры диалога

Применена **смешанная структура**: иерархическое **меню** (навигация MoonShine) и **экранные формы** (CRUD-страницы ресурсов). Для посетителя витрины — упрощённый сценарий просмотра без авторизации.

### 2.3.2 Иерархия меню административной панели

Меню задаётся в `app/MoonShine/Layouts/MoonShineLayout.php`:

| Группа / пункт | Ресурс | Видимость |
|----------------|--------|-----------|
| Система (только админ) | Пользователи, Роли, Отчёты | `isSuperUser()` |
| Кредитные заявки | `CreditApplicationResource` | По правам |
| Клиенты | `ClientResource` | По правам |
| Доп. опции | `OptionResource` | По правам |
| Автомобили → Новые | `CarNewResource` | По правам |
| Автомобили → С пробегом | `CarUsedResource` | По правам |
| Сделки → Продажа | `SailSellResource` | По правам |
| Сделки → Покупка | `SailBuyResource` | По правам |

### 2.3.3 Сценарий «Оформление продажи автомобиля»

| Шаг | Действие пользователя | Отклик системы |
|-----|----------------------|----------------|
| 1 | Вход на `/admin` | Форма авторизации MoonShine |
| 2 | Меню «Сделки» → «Продажа» | Список сделок продажи |
| 3 | «Создать» | Форма: клиент, автомобиль (только IN_STOCK), цена, опции, файлы |
| 4 | Статус «Подготовка» | `CarStatus::RESERVED` |
| 5 | Статус «Завершена» | `CarStatus::SOLD`, кэш каталога сброшен |
| 6 | Статус «Отменена» | `CarStatus::IN_STOCK` |

Согласованное изменение статусов автомобиля и сделки показано на **рисунке 2.4** (`diagrams/state-sail-car.mmd`).

### 2.3.4 Сценарий «Просмотр каталога посетителем»

| Шаг | Действие | Отклик |
|-----|----------|--------|
| 1 | GET `/` | `HomeController@index` |
| 2 | Переключение вкладок «Новый» / «Б/У» | Фильтр по `CarType`, только `IN_STOCK` |
| 3 | Просмотр карточки | Марка, модель, цена, фото |

**Вывод по главе 2.** Спроектированы DFD, трёхуровневая модель БД и сценарии диалога, полностью соответствующие архитектуре DriveLine.
