# Введение

Современный автомобильный рынок характеризуется высокой конкуренцией, широким ассортиментом транспортных средств и необходимостью оперативного учёта складских остатков, клиентской базы и сделок. Автосалоны ежедневно обрабатывают значительные объёмы информации: карточки автомобилей с уникальными идентификаторами (VIN), договоры купли-продажи, данные покупателей, заявки на кредитование, отчёты для руководства. При ручном или разрозненном учёте (электронные таблицы, бумажные журналы) возрастает риск ошибок в статусах автомобилей на складе, дублирования записей о клиентах, задержек при формировании отчётности.

**Актуальность** выбранной темы обусловлена потребностью автосалона в единой информационной системе, которая объединяет публичный каталог для посетителей и закрытый бэк-офис для сотрудников, автоматически синхронизирует склад при изменении статуса сделки и поддерживает разграничение прав доступа. Разработка такой системы в рамках курса «Проектирование информационных систем» позволяет применить на практике методы структурного анализа, проектирования баз данных и проектирования пользовательского интерфейса.

**Объект исследования** — деятельность автосалона по учёту автомобилей, клиентов и сделок.

**Предмет исследования** — процессы сбора, хранения, обработки и представления информации в автоматизированной информационной системе DriveLine.

**Цель работы** — спроектировать и реализовать информационную систему учёта автомобилей, клиентов и сделок автосалона с публичной витриной и административной панелью.

Для достижения цели поставлены следующие **задачи**:

1. Провести анализ предметной области автосалона и системный анализ функций, подлежащих автоматизации.
2. Обосновать экономическую и техническую целесообразность применения вычислительной техники.
3. Сформулировать постановку задачи и требования к проектируемой системе.
4. Проанализировать существующие программные решения и обосновать выбор технологий проектирования и разработки.
5. Разработать информационную модель (DFD) и модели базы данных (концептуальную, логическую и физическую).
6. Спроектировать сценарий диалога пользователя с системой.
7. Реализовать информационную систему на выбранном программно-аппаратном стеке.
8. Провести тестирование и описать технологию эксплуатации.

**Методы исследования:** структурный анализ предметной области (IDEF0), моделирование потоков данных (DFD), ER-моделирование (нотация IDEF1X), нормализация реляционных отношений, объектно-ориентированная разработка, функциональное и модульное тестирование.

**Практическая значимость** работы состоит в создании работоспособного веб-приложения DriveLine, развёртываемого в Docker, с админ-панелью MoonShine, реляционной СУБД PostgreSQL, фоновой обработкой отчётов и кредитных заявок через очереди Redis.

**Структура пояснительной записки.** Работа состоит из введения, трёх глав, заключения, списка использованных источников и приложений. В первой главе выполнен анализ предметной области и постановка задачи. Во второй — проектирование информационной модели, базы данных и диалога. В третьей — описание реализации, программной архитектуры и тестирования. В приложениях приведены диаграммы IDEF0, DFD, ER-модели, сценарии диалога, таблицы полей БД, листинги программных модулей и инструкция по развёртыванию.

**Краткий обзор источников.** При выполнении работы использованы методические рекомендации кафедры по курсовым работам по дисциплине «Проектирование ИС», учебная литература по проектированию баз данных и информационных систем, официальная документация Laravel, MoonShine и PostgreSQL, а также материалы по CRM-системам для автобизнеса.
# 1 Аналитическая часть

## 1.1 Анализ предметной области

### 1.1.1 Технико-экономическая характеристика предметной области

В качестве предметной области выступает автосалон — коммерческая организация, основной целью которой является реализация новых и подержанных автомобилей, сопутствующее обслуживание (дополнительные опции) и сопровождение сделок с привлечением кредитных продуктов.

**Организационная структура (упрощённо):**

- руководитель (администратор системы, доступ к отчётам и управлению пользователями);
- менеджеры по продажам (оформление сделок, работа с клиентами, кредитные заявки);
- специалисты по закупке (приём автомобилей на склад — сделки типа «покупка»).

**Основные бизнес-процессы:**

1. Учёт автомобилей на складе (марка, модель, VIN, цена, пробег, тип «новый/Б/У», статус наличия).
2. Ведение реестра клиентов (контакты, паспортные данные).
3. Оформление сделок продажи и закупки с жизненным циклом статусов.
4. Подключение дополнительных услуг к сделке.
5. Рассмотрение заявок на кредитование.
6. Формирование периодических отчётов по завершённым сделкам.
7. Публикация каталога доступных автомобилей для посетителей сайта.

**Показатели функционирования (ориентировочные для среднего автосалона):**

| Показатель | Оценка |
|------------|--------|
| Число единиц на складе | 50–300 автомобилей |
| Сделок в месяц | 30–150 |
| Средний чек сделки | 1,5–4,0 млн руб. |
| Доля сделок с кредитом | 20–40 % |
| Число сотрудников бэк-офиса | 5–25 |

### 1.1.2 Системный анализ объекта исследования

Для анализа применён **структурный подход** и методология **SADT (IDEF0)**. Функциональная модель отражает декомпозицию деятельности автосалона независимо от организационной структуры.

**Контекстная диаграмма A-0** (рисунок 1.1, исходник — `diagrams/idef0-context.mmd`) описывает систему «Управление деятельностью автосалона»:

- **Входы:** заявки клиентов, поступление автомобилей, прайс-листы и регламенты.
- **Управление:** законодательство, внутренние регламенты, ценовая политика.
- **Механизмы:** персонал, информационная система DriveLine.
- **Выходы:** оформленные сделки и договоры, отчёты для руководства, актуальное состояние склада.

**Декомпозиция первого уровня** (рисунок 1.2, `diagrams/idef0-decomposition.mmd`) выделяет подсистемы:

| Код | Функция | Соответствие в DriveLine |
|-----|---------|--------------------------|
| A1 | Учёт автомобилей на складе | `CarNewResource`, `CarUsedResource`, `CarService` |
| A2 | Ведение базы клиентов | `ClientResource` |
| A3 | Оформление сделок | `SailBuyResource`, `SailSellResource`, `SailService` |
| A4 | Учёт дополнительных опций | `OptionResource`, связь `option_sale` |
| A5 | Обработка кредитных заявок | `CreditApplicationResource`, очередь |
| A6 | Формирование отчётности | `ReportResource`, `ReportService` |
| A7 | Публичный каталог | `HomeController`, витрина Blade |

Между подсистемами установлены информационные связи: данные о клиентах и автомобилях используются при оформлении сделок; результаты сделок агрегируются в отчётах; склад отображается в публичном каталоге.

**Существующая технология («as-is»).** До внедрения ИС типичны следующие средства: журнал учёта на бумаге или в Excel, разрозненные файлы с карточками клиентов, ручное обновление статуса автомобиля при продаже, отчёты формируются вручную в конце месяца.

**Недостатки существующей практики:**

| Недостаток | Проявление | Устранение в проекте |
|------------|------------|----------------------|

| Слабая защита данных | Общий доступ к файлам | Роли и `user_permissions` в MoonShine |

## 2.2 Проектирование информационного хранилища

Целевая модель данных — **реляционная**, СУБД **PostgreSQL 15**, нормальная форма — **3НФ**.

### 2.2.1 Концептуальная модель БД

Концептуальная ER-диаграмма (рисунок 2.3, `diagrams/er-conceptual.mmd`) включает сущности:

- `cars` — автомобили на складе;
- `clients` — покупатели;
- `sails` — сделки (покупка/продажа);
- `options` — дополнительные услуги;
- связь **N:M** `sails`–`options` реализуется таблицей `option_sale`;
- `credit_applications`, `reports`, `users`, `user_roles`, `user_permissions`.

### 2.2.2 Логическая модель БД

**Приведение к 3НФ.**

1. Связь «сделка — опция» N:M устранена введением ассоциативной таблицы `option_sale` (две связи 1:N).
2. Атрибуты клиента, автомобиля и менеджера не дублируются в `sails` — только внешние ключи `client_id`, `car_id`, `user_id`.
3. Права пользователя вынесены в отдельную сущность `user_permissions` (JSON), что исключает повторение матрицы прав для каждой операции.

**Таблица 2.1 — Примеры транзакций**

| Транзакция | Затрагиваемые таблицы | Ограничение целостности |
|------------|----------------------|-------------------------|
| Завершить сделку продажи | `sails`, `cars` | Статус авто → SOLD; сделка COMPLETED |
| Отменить сделку продажи | `sails`, `cars` | Статус авто → IN_STOCK |
| Удалить клиента | `clients`, `credit_applications` | CASCADE по заявкам |
| Удалить сделку | `sails`, `option_sale` | CASCADE по связям опций |
| Сформировать отчёт | `reports`, `sails` | Только COMPLETED за период |

**Первичные и внешние ключи** (фрагмент):

- `sails.id` — PK; `client_id`, `user_id`, `car_id` — FK с `ON DELETE SET NULL` (миграция `create_sails_table`);
- `option_sale` — составной PK (`option_id`, `sail_id`);
- `clients.phone` — UNIQUE.

Физическая модель соответствует миграциям Laravel в каталоге `database/migrations/`. Полное описание полей — в **приложении Е**.

| Поле | Тип PostgreSQL | Ограничения | Примечание |

| id | BIGSERIAL | PK | Суррогатный ключ |

| model | VARCHAR | NOT NULL, INDEX | Модель |

**Эволюция модели:** ранее использовалось поле `count` (количество экземпляров); заменено на `status` (миграция `add_cars_status`) — одна запись соответствует одному физическому автомобилю.

**Защита данных на уровне приложения:**

- роли: `user_roles` (администратор `id=1`, менеджер `id=2`);
- детальные права: JSON в `user_permissions.permissions` по ресурсам MoonShine;

## 3.5 Технология обработки информации

**Сценарий «Продажа автомобиля»:**

1. Менеджер создаёт сделку типа SELL, выбирает клиента и автомобиль со статусом IN_STOCK.
2. `SailService::createSellSail` в транзакции проверяет доступность авто.
3. При статусе «Подготовка» вызывается `syncSellCarStatus` → RESERVED.
4. При «Завершена» → SOLD; `CarService::forgetCarsCache()` обновляет витрину.
5. При «Отменена» → IN_STOCK.

**Фоновые процессы:** `php artisan queue:work` / Horizon обрабатывает `GenerateReportJob` и заявки на кредит.

**Развёртывание** (кратко): `cp .env.example .env`, `make up`, `npm run build` — подробно в приложении И и [README.md](../../README.md).

# Заключение

В ходе выполнения курсовой работы по дисциплине «Проектирование информационных систем» спроектирована и реализована информационная система **DriveLine** для автоматизации учёта автомобилей, клиентов и сделок автосалона.

В **аналитической части** рассмотрена предметная область автосалона, построена функциональная модель в нотации IDEF0, обоснована необходимость применения вычислительной техники, сформулирована постановка задачи и проведён сравнительный анализ существующих программных решений.

В **проектной части** разработаны диаграммы потоков данных, концептуальная, логическая и физическая модели базы данных в третьей нормальной форме, а также сценарии диалога для административной панели MoonShine и публичной витрины.

В **части реализации** создано веб-приложение на стеке Laravel 13, PostgreSQL, MoonShine и Redis с разделением на публичный каталог и закрытый бэк-офис, сервисным слоем бизнес-логики и фоновой генерацией отчётов. Ключевое бизнес-правило — автоматическая синхронизация статуса автомобиля при изменении статуса сделки продажи — реализовано в модуле `SailService`.

**Практическая ценность:** снижение риска ошибок складского учёта, ускорение подготовки отчётности, централизованное хранение данных о клиентах и сделках, разграничение прав доступа сотрудников.

**Перспективы развития:** REST API для мобильного приложения; онлайн-заявка с витрины; интеграция с банковскими API по кредитам; расширение автотестов бизнес-логики; доработка аналитической панели для руководителя.
