# Введение

Современный автомобильный рынок характеризуется высокой конкуренцией, широким ассортиментом транспортных средств и необходимостью оперативного учёта складских остатков, клиентской базы и сделок. Автосалоны ежедневно обрабатывают значительные объёмы информации: карточки автомобилей с уникальными идентификаторами (VIN), договоры купли-продажи, данные покупателей, заявки на кредитование, отчёты для руководства. При ручном или разрозненном учёте (электронные таблицы, бумажные журналы) возрастает риск ошибок в статусах автомобилей на складе, дублирования записей о клиентах, задержек при формировании отчётности.

**Актуальность** выбранной темы обусловлена потребностью автосалона в единой информационной системе, которая объединяет публичный каталог для посетителей и закрытый бэк-офис для сотрудников, автоматически синхронизирует склад при изменении статуса сделки и поддерживает разграничение прав доступа. Разработка такой системы в рамках курса «Проектирование информационных систем» позволяет применить на практике методы структурного анализа, проектирования баз данных и проектирования пользовательского интерфейса.

**Объект исследования** — деятельность автосалона по учёту автомобилей, клиентов и сделок.

**Предмет исследования** — процессы сбора, хранения, обработки и представления информации в автоматизированной информационной системе DriveLine.

**Цель работы** — спроектировать и реализовать информационную систему учёта автомобилей, клиентов и сделок автосалона с публичной витриной и административной панелью.

Для достижения цели поставлены следующие **задачи**:

1. Провести анализ предметной области автосалона и системный анализ функций, подлежащих автоматизации.
2. Обосновать экономическую и техническую целесообразность применения вычислительной техники.
3. Сформулировать постановку задачи и требования к проектируемой системе.
4. Проанализировать существующие программные решения и обосновать выбор технологий проектирования и разработки.
5. Разработать информационную модель (DFD) и модели базы данных (концептуальную, логическую и физическую).
6. Спроектировать сценарий диалога пользователя с системой.
7. Реализовать информационную систему на выбранном программно-аппаратном стеке.
8. Провести тестирование и описать технологию эксплуатации.

**Методы исследования:** структурный анализ предметной области (IDEF0), моделирование потоков данных (DFD), ER-моделирование (нотация IDEF1X), нормализация реляционных отношений, объектно-ориентированная разработка, функциональное и модульное тестирование.

**Практическая значимость** работы состоит в создании работоспособного веб-приложения DriveLine, развёртываемого в Docker, с админ-панелью MoonShine, реляционной СУБД PostgreSQL, фоновой обработкой отчётов и кредитных заявок через очереди Redis.

**Структура пояснительной записки.** Работа состоит из введения, трёх глав, заключения, списка использованных источников и приложений. В первой главе выполнен анализ предметной области и постановка задачи. Во второй — проектирование информационной модели, базы данных и диалога. В третьей — описание реализации, программной архитектуры и тестирования. В приложениях приведены диаграммы IDEF0, DFD, ER-модели, сценарии диалога, таблицы полей БД, листинги программных модулей и инструкция по развёртыванию.

**Краткий обзор источников.** При выполнении работы использованы методические рекомендации кафедры по курсовым работам по дисциплине «Проектирование ИС», учебная литература по проектированию баз данных и информационных систем, официальная документация Laravel, MoonShine и PostgreSQL, а также материалы по CRM-системам для автобизнеса.
# 1 Аналитическая часть

## 1.1 Анализ предметной области

### 1.1.1 Технико-экономическая характеристика предметной области

В качестве предметной области выступает автосалон — коммерческая организация, основной целью которой является реализация новых и подержанных автомобилей, сопутствующее обслуживание (дополнительные опции) и сопровождение сделок с привлечением кредитных продуктов.

**Организационная структура (упрощённо):**

- руководитель (администратор системы, доступ к отчётам и управлению пользователями);
- менеджеры по продажам (оформление сделок, работа с клиентами, кредитные заявки);
- специалисты по закупке (приём автомобилей на склад — сделки типа «покупка»).

**Основные бизнес-процессы:**

1. Учёт автомобилей на складе (марка, модель, VIN, цена, пробег, тип «новый/Б/У», статус наличия).
2. Ведение реестра клиентов (контакты, паспортные данные).
3. Оформление сделок продажи и закупки с жизненным циклом статусов.
4. Подключение дополнительных услуг к сделке.
5. Рассмотрение заявок на кредитование.
6. Формирование периодических отчётов по завершённым сделкам.
7. Публикация каталога доступных автомобилей для посетителей сайта.

**Показатели функционирования (ориентировочные для среднего автосалона):**

| Показатель | Оценка |
|------------|--------|
| Число единиц на складе | 50–300 автомобилей |
| Сделок в месяц | 30–150 |
| Средний чек сделки | 1,5–4,0 млн руб. |
| Доля сделок с кредитом | 20–40 % |
| Число сотрудников бэк-офиса | 5–25 |

### 1.1.2 Системный анализ объекта исследования

Для анализа применён **структурный подход** и методология **SADT (IDEF0)**. Функциональная модель отражает декомпозицию деятельности автосалона независимо от организационной структуры.

**Контекстная диаграмма A-0** (рисунок 1.1, исходник — `diagrams/idef0-context.mmd`) описывает систему «Управление деятельностью автосалона»:

- **Входы:** заявки клиентов, поступление автомобилей, прайс-листы и регламенты.
- **Управление:** законодательство, внутренние регламенты, ценовая политика.
- **Механизмы:** персонал, информационная система DriveLine.
- **Выходы:** оформленные сделки и договоры, отчёты для руководства, актуальное состояние склада.

**Декомпозиция первого уровня** (рисунок 1.2, `diagrams/idef0-decomposition.mmd`) выделяет подсистемы:

| Код | Функция | Соответствие в DriveLine |
|-----|---------|--------------------------|
| A1 | Учёт автомобилей на складе | `CarNewResource`, `CarUsedResource`, `CarService` |
| A2 | Ведение базы клиентов | `ClientResource` |
| A3 | Оформление сделок | `SailBuyResource`, `SailSellResource`, `SailService` |
| A4 | Учёт дополнительных опций | `OptionResource`, связь `option_sale` |
| A5 | Обработка кредитных заявок | `CreditApplicationResource`, очередь |
| A6 | Формирование отчётности | `ReportResource`, `ReportService` |
| A7 | Публичный каталог | `HomeController`, витрина Blade |

Между подсистемами установлены информационные связи: данные о клиентах и автомобилях используются при оформлении сделок; результаты сделок агрегируются в отчётах; склад отображается в публичном каталоге.

## 1.2 Обоснование необходимости и цели использования вычислительной техники

**Существующая технология («as-is»).** До внедрения ИС типичны следующие средства: журнал учёта на бумаге или в Excel, разрозненные файлы с карточками клиентов, ручное обновление статуса автомобиля при продаже, отчёты формируются вручную в конце месяца.

**Недостатки существующей практики:**

| Недостаток | Проявление | Устранение в проекте |
|------------|------------|----------------------|
| Низкая оперативность | Задержка актуализации склада | Автоматическая смена `CarStatus` при смене статуса сделки |
| Ошибки учёта | Авто числится «в наличии» после продажи | `SailService::syncSellCarStatus` |
| Дублирование данных | Повторный ввод клиента | Единая таблица `clients`, уникальный `phone` |
| Высокая трудоёмкость отчётов | Ручная сводка в Excel | `GenerateReportJob`, экспорт XLSX |
| Слабая защита данных | Общий доступ к файлам | Роли и `user_permissions` в MoonShine |

**Таблица 1.1 — Потоки информации (оценочно)**

| Документ / поток | Источник | Получатель | Периодичность | Объём, стр./записей в год |
|------------------|----------|------------|---------------|---------------------------|
| Карточка автомобиля | Менеджер, парсер | Склад, витрина | По факту поступления | 200–500 |

| Отчёт по продажам | ИС | Руководитель | 1–4/мес. | 12–48 |

**Экономическая целесообразность:** сокращение времени формирования отчёта с нескольких часов до минут (фоновая генерация), снижение числа ошибок статуса склада, единая точка ввода данных для менеджеров.

# 2 Проектная часть

Проектная часть описывает решения по вертикали проектирования: информационные потоки, структуру хранилища данных и организацию диалога пользователя с системой DriveLine. Все сущности согласованы с аналитической частью (глава 1) и реализацией (глава 3).

## 2.1 Разработка информационной модели

Информационная модель представлена **диаграммами потоков данных (DFD)** по методологии Гейна–Сарсона.

### 2.1.1 Контекстная DFD

На рисунке 2.1 (исходник `diagrams/dfd-context.mmd`) система **DriveLine** (процесс 0) показана в окружении внешних сущностей:

- **Менеджер** — ведёт клиентов, автомобили, сделки, кредитные заявки;
- **Администратор** — управляет пользователями, ролями, формирует отчёты;

## 2.2 Проектирование информационного хранилища

Целевая модель данных — **реляционная**, СУБД **PostgreSQL 15**, нормальная форма — **3НФ**.

### 2.2.1 Концептуальная модель БД

Концептуальная ER-диаграмма (рисунок 2.3, `diagrams/er-conceptual.mmd`) включает сущности:

- `cars` — автомобили на складе;
- `clients` — покупатели;
- `sails` — сделки (покупка/продажа);
- `options` — дополнительные услуги;
- связь **N:M** `sails`–`options` реализуется таблицей `option_sale`;
- `credit_applications`, `reports`, `users`, `user_roles`, `user_permissions`.

### 2.2.2 Логическая модель БД

Логическая модель строится на основе концептуальной (рисунок 2.3) с учётом принимаемой реляционной модели и приводится к третьей нормальной форме. Ниже последовательно рассмотрены: связи между сущностями, нормализация, проверка транзакций и поддержка ссылочной целостности.

| `user_roles` | `users` | 1 : N | неидентифицирующая | необязательная | SET NULL |
| `users` | `user_permissions` | 1 : 1 | идентифицирующая | обязательная | CASCADE |

Исходная семантическая связь N : M между сущностями `sails` и `options` (одна сделка содержит несколько дополнительных опций, одна опция применима к нескольким сделкам) устранена путём введения ассоциативной сущности `option_sale` с составным первичным ключом (`sail_id`, `option_id`), что соответствует требованиям реляционной модели.

**Приведение к нормальным формам.**

1. **Первая нормальная форма (1НФ).** Все атрибуты сущностей являются атомарными: `cars.vin_code`, `clients.phone`, `cars.year`, `sails.price` хранят одно значение на запись. Поля с потенциально составным содержимым (`cars.images`, `sails.files`, `user_permissions.permissions`) вынесены в JSON-домен СУБД PostgreSQL, что допустимо в рамках 1НФ при условии, что прикладная логика не оперирует подэлементами как отдельными атрибутами реляционной схемы.
2. **Вторая нормальная форма (2НФ).** Все сущности с составным первичным ключом сведены к одной ассоциативной таблице `option_sale`, в которой нет неключевых атрибутов, поэтому условие полной функциональной зависимости неключевых атрибутов от первичного ключа выполняется тривиально. Во всех остальных таблицах первичным ключом является суррогатный `id` (BIGSERIAL), и каждый неключевой атрибут функционально зависит только от него.
3. **Третья нормальная форма (3НФ).** Транзитивные зависимости устранены:
   - роль пользователя вынесена из таблицы `users` в отдельную сущность `user_roles` (атрибут `users.user_role_id` заменяет дублирование наименования роли);
   - детальные права доступа вынесены из `users` в сущность `user_permissions` (JSON), что исключает повторение матрицы прав для каждой операции;
   - данные клиента, автомобиля и менеджера не дублируются в `sails` — только внешние ключи `client_id`, `car_id`, `user_id`;
   - в таблице `reports` хранятся только параметры запроса и ссылка на сгенерированный файл; сами строки отчёта не материализуются в БД, а вычисляются из `sails` при формировании XLSX (производный показатель не хранится).

**Проверка выполнения транзакций.** Корректность принятой логической модели подтверждена прогоном через типовые транзакции системы DriveLine (таблица 2.2).

**Таблица 2.2 — Контроль выполнения транзакций над логической моделью**

| Транзакция | Тип SQL-операции | Затрагиваемые таблицы | Ограничение целостности |
|------------|------------------|----------------------|-------------------------|
| Регистрация клиента | INSERT | `clients` | UNIQUE по `phone` |
| Постановка автомобиля на склад (закупка) | INSERT, UPDATE | `cars`, `sails` | `cars.status = IN_STOCK` по факту завершения сделки |
| Создание сделки продажи в статусе «Подготовка» | INSERT, UPDATE | `sails`, `cars` | `cars.status = RESERVED`; авто доступно только при IN_STOCK |
| Завершение сделки продажи | UPDATE | `sails`, `cars` | `cars.status = SOLD`; `sails.status = COMPLETED` |
| Отмена сделки продажи | UPDATE | `sails`, `cars` | `cars.status = IN_STOCK` |
| Удаление сделки | DELETE | `sails`, `option_sale` | CASCADE по `option_sale` |
| Подача заявки на кредит | INSERT | `credit_applications` | FK `client_id` обязателен |
| Удаление клиента | DELETE | `clients`, `credit_applications`, `sails` | CASCADE по заявкам, SET NULL в `sails.client_id` |
| Формирование отчёта за период | INSERT, SELECT | `reports`, `sails` | Выборка только `sails.status = COMPLETED` и `sails.type` соответствует типу отчёта |
| Удаление пользователя | DELETE | `users`, `user_permissions`, `sails`, `reports`, `credit_applications` | CASCADE по `user_permissions`, SET NULL по остальным |

**Поддержка ссылочной целостности.** Для каждой связи определён тип триггера на действие `DELETE` родительской записи в соответствии с типологией методички (RESTRICT / CASCADE / SET NULL / SET DEFAULT / NO CHECK). Выбор обоснован требованиями предметной области (таблица 2.3).

**Таблица 2.3 — Ссылочная целостность и обоснование триггеров ON DELETE**

| Связь (родитель → дочерний.FK) | Тип триггера | Обоснование |
|--------------------------------|--------------|-------------|
| `clients.id` → `sails.client_id` | SET NULL | История сделок должна сохраняться даже при удалении клиента |
| `users.id` → `sails.user_id` | SET NULL | Сделка остаётся в системе при увольнении менеджера |
| `cars.id` → `sails.car_id` | SET NULL | Журнал сделок не должен разрушаться при удалении карточки авто; для физического сохранения карточки применяется soft delete (`cars.deleted_at`) |
| `sails.id` → `option_sale.sail_id` | CASCADE | Связь «опция в сделке» лишена смысла без самой сделки |
| `options.id` → `option_sale.option_id` | CASCADE | При удалении опции из справочника соответствующие привязки также удаляются |
| `clients.id` → `credit_applications.client_id` | CASCADE | Заявка персонализирована; без клиента не имеет смысла хранить её данные |
| `users.id` → `credit_applications.user_id` | SET NULL | Заявка остаётся в системе при увольнении менеджера |
| `users.id` → `reports.user_id` | SET NULL | Сгенерированные отчёты остаются в системе |
| `user_roles.id` → `users.user_role_id` | SET NULL | Пользователь может временно остаться без роли (будет назначена администратором) |
| `users.id` → `user_permissions.user_id` | CASCADE | Запись прав не имеет смысла без пользователя |

Дополнительно действуют ограничения уровня атрибутов: `NOT NULL` для всех обязательных полей (`cars.vin_code`, `cars.mark`, `clients.name`, `sails.type` и др.), `DEFAULT` для `cars.status = 1` и `sails.status = PENDING`, ограничение `UNIQUE(phone)` в `clients`. Бизнес-правило «при завершении сделки продажи статус автомобиля должен синхронизироваться» реализовано на уровне приложения в сервисе `SailService::syncSellCarStatus` (см. главу 3), что соответствует подходу методички «реализация бизнес-правил средствами приложения» при отсутствии CHECK-ограничений в выбранной СУБД.

**Первичные и внешние ключи** (сводный фрагмент):

- `sails.id` — PK; `client_id`, `user_id`, `car_id` — FK с `ON DELETE SET NULL` (миграция `create_sails_table`);
- `option_sale` — составной PK (`option_id`, `sail_id`); обе FK с `CASCADE`;
- `credit_applications.client_id` — FK с `CASCADE`; `user_id` — FK с `SET NULL`;
- `clients.phone` — UNIQUE;
- `cars.deleted_at` — атрибут soft delete (физическое удаление не выполняется).

**Таблица 2.4 — Фрагмент физической модели (таблица `cars`)**

| Поле | Тип PostgreSQL | Ограничения | Примечание |
|------|----------------|-------------|------------|
| id | BIGSERIAL | PK | Суррогатный ключ |
| mark | VARCHAR | NOT NULL, INDEX | Марка |
| model | VARCHAR | NOT NULL, INDEX | Модель |

| color | ENUM | NOT NULL | `CarColor` |

| preview, images | VARCHAR / JSON | NULL | Медиа |
| deleted_at | TIMESTAMP | NULL | Soft delete |

**Контролируемая денормализация (анализ необходимости).** В физическую модель не внесено избыточных производных атрибутов: общая сумма сделки с учётом опций вычисляется на лету при формировании отчёта (`ReportService`), а не хранится в `sails`. Это соответствует требованию методички взвешивать стоимость хранения производных данных против стоимости их вычисления; при текущих объёмах (30–150 сделок в месяц) фактор хранения проигрывает фактору простоты согласования. Служебные таблицы для перечислений (`CarColor`, `CarType`, `SailStatus`, `SailType`, `ReportStatus`) реализованы как backed-enum в PHP с приведением к нативным типам PostgreSQL `ENUM` / `SMALLINT`, что устраняет рассогласованные значения без введения отдельных справочных таблиц.

**Эволюция модели:** ранее использовалось поле `count` (количество экземпляров); заменено на `status` (миграция `add_cars_status`) — одна запись соответствует одному физическому автомобилю, что упрощает синхронизацию статуса со сделкой.

**Защита данных на уровне приложения:**

- роли: `user_roles` (администратор `id=1`, менеджер `id=2`);
- детальные права: JSON в `user_permissions.permissions` по ресурсам MoonShine;
- суперпользователь (`isSuperUser`) — полный доступ к отчётам и пользователям (`MoonShineServiceProvider`).

### 3.4.2 Описание программных модулей и алгоритмы

Ниже приведены пошаговые описания алгоритмов (блок-схемы) основных расчётных модулей. Графические блок-схемы — рисунки 3.2 и 3.3, исходники — в каталоге `diagrams/`. Листинги соответствующих PHP-классов размещены в **приложении Ж**.

**Алгоритм 1. Синхронизация статуса автомобиля при сделке продажи** (рисунок 3.2, `diagrams/algorithm-sail-sync.mmd`; реализация — `App\Services\SailService::syncSellCarStatus`, файл `app/Services/SailService.php`):

1. Получить объект сделки `Sail` и желаемое новое значение `SailStatus`.
2. Если `Sail::type ≠ SELL` (сделка не является продажей) — завершить алгоритм без действий.
3. Если `Sail::car_id` пуст (автомобиль не привязан) — завершить алгоритм без действий.
4. Сопоставить новое состояние сделки и состояние автомобиля по таблице соответствия:
   - `SailStatus::PENDING` → `CarStatus::RESERVED` (резерв);
   - `SailStatus::COMPLETED` → `CarStatus::SOLD` (продан);
   - `SailStatus::CANCELLED` → `CarStatus::IN_STOCK` (возврат в наличие);
   - прочее — алгоритм завершается без действий.
5. Вызвать `CarService::updateCarStatus(carId, carStatus)`, который в транзакции обновляет `cars.status` и сбрасывает кэш публичного каталога (`CarService::forgetCarsCache()`).
6. Завершение алгоритма.

При создании сделки продажи дополнительно выполняется предварительный контроль (`SailService::createSellSail`): шаги «Проверить, что автомобиль существует» и «Проверить, что `cars.status = IN_STOCK`»; при невыполнении любого условия — выбрасывается исключение и транзакция откатывается.

**Алгоритм 2. Формирование отчёта по завершённым сделкам** (рисунок 3.3, `diagrams/algorithm-report.mmd`; реализация — `App\Services\ReportService::generate` + `App\Jobs\GenerateReportJob`):

1. Менеджер создаёт сущность `Report` с параметрами: период `from–to`, тип отчёта (`SailType::SELL` или `SailType::BUY`).
2. `ReportService::create` сохраняет запись со статусом `ReportStatus::PENDING` и помещает в очередь задание `GenerateReportJob`.
3. Воркер очереди (`php artisan queue:work` или Horizon) принимает задание и вызывает `ReportService::generate(Report)`.
4. Статус отчёта переводится в `ReportStatus::PROCESSING`.
5. Выполняется выборка `fetchCompletedSails`: из таблицы `sails` отбираются записи с условиями `status = COMPLETED`, `type = report.type` и `created_at` в интервале `[from 00:00:00; to 23:59:59]`.
6. Метод `buildSpreadsheet` создаёт XLSX-файл через OpenSpout:
   - формируется относительный путь `reports/{type}_{id}_{ts}.xlsx`;
   - записывается строка заголовков: «№», «Дата продажи/покупки», «Автомобиль», «VIN», «Сумма», «Клиент», «Менеджер»;
   - в цикле для каждой сделки добавляется строка и накапливается итоговая сумма;
   - в конце добавляется строка «Итого: …» с общей суммой.
7. При успешном завершении `Report::status = COMPLETED` и сохраняется относительный путь `Report::file`. При исключении — `Report::status = FAILED`, исключение перебрасывается выше для логирования.
8. Менеджер скачивает готовый файл по ссылке в карточке отчёта.

## 3.5 Технология обработки информации

**Сценарий «Продажа автомобиля»:**

1. Менеджер создаёт сделку типа SELL, выбирает клиента и автомобиль со статусом IN_STOCK.
2. `SailService::createSellSail` в транзакции проверяет доступность авто.
3. При статусе «Подготовка» вызывается `syncSellCarStatus` → RESERVED.
4. При «Завершена» → SOLD; `CarService::forgetCarsCache()` обновляет витрину.
5. При «Отменена» → IN_STOCK.

**Фоновые процессы:** `php artisan queue:work` / Horizon обрабатывает `GenerateReportJob` и заявки на кредит.

**Развёртывание** (кратко): `cp .env.example .env`, `make up`, `npm run build` — подробно в приложении И и [README.md](../../README.md).

# Заключение

В ходе выполнения курсовой работы по дисциплине «Проектирование информационных систем» спроектирована и реализована информационная система **DriveLine** для автоматизации учёта автомобилей, клиентов и сделок автосалона.

В **аналитической части** рассмотрена предметная область автосалона, построена функциональная модель в нотации IDEF0, обоснована необходимость применения вычислительной техники, сформулирована постановка задачи и проведён сравнительный анализ существующих программных решений.

В **проектной части** разработаны диаграммы потоков данных, концептуальная, логическая и физическая модели базы данных в третьей нормальной форме, а также сценарии диалога для административной панели MoonShine и публичной витрины.

В **части реализации** создано веб-приложение на стеке Laravel 13, PostgreSQL, MoonShine и Redis с разделением на публичный каталог и закрытый бэк-офис, сервисным слоем бизнес-логики и фоновой генерацией отчётов. Ключевое бизнес-правило — автоматическая синхронизация статуса автомобиля при изменении статуса сделки продажи — реализовано в модуле `SailService`.

**Практическая ценность:** снижение риска ошибок складского учёта, ускорение подготовки отчётности, централизованное хранение данных о клиентах и сделках, разграничение прав доступа сотрудников.

**Перспективы развития:** REST API для мобильного приложения; онлайн-заявка с витрины; интеграция с банковскими API по кредитам; расширение автотестов бизнес-логики; доработка аналитической панели для руководителя.
